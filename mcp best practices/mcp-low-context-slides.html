Engineering Reference · MCP Architecture

Keeping MCPs
Low-Context

Five architectural patterns to prevent tool-manifest bloat from overwhelming your agent's context window — without sacrificing capability.

context window usage 94%
The Problem

Naive MCP Integration Scales Badly

As tool catalogs grow, the default approach — inject every registered MCP server manifest on every turn — creates a compounding cost problem.

Each manifest costs hundreds of tokens. 30 tools × 300 tokens = 9,000 tokens before the user says anything. At 100 tools, you're burning most of a context window on plumbing.

The result: slower responses, confused routing, higher cost, and a hard ceiling on how many capabilities you can register.

# What naive loading looks like # on every single turn: tools = [ "create_user", # 280 tok "delete_user", # 260 tok "list_tenants", # 310 tok "query_metrics", # 340 tok "run_playwright", # 890 tok "fetch_figma_doc", # 720 tok "search_confluence", # 480 tok # ... 23 more ... ] # Total before turn 1: ~8,400 tokens # Relevant tools used: 1–2
Five Patterns

A Layered Defense Against Context Bloat

PATTERN 01
Expose Minimal Surface
Split large MCP servers into small, semantically cohesive units. ≤4 servers per software component.
PATTERN 02
Lazy Tool Loading
Inject a single ToolSearch tool on turn 1. Manifests enter context only when the agent explicitly requests them.
PATTERN 03
Semantic Router / Registry
Back discovery with a vector index. Route queries to the top-k most relevant tools — not everything registered.
PATTERN 04
Higher-Level Abstractions
Replace clusters of fine-grained tools with Skills, CLI bridges, or Subagents that encapsulate the complexity.
PATTERN 05
Per-Session Tool Sets
Maintain multiple mcp.*.json configs and select the right one per project. Zero irrelevant tools in context.
Pattern 01 · Minimal Surface

Split Large MCP Servers

A single server with 40 tools injects all 40 on every turn. Break it apart by semantic domain — each server owns exactly one capability area.

  1. Audit your MCP server and list all exposed tools.
  2. Group by the noun they operate on or the workflow they belong to.
  3. Create a separate server binary per group (≤4 per component).
  4. Update client configs to reference the new servers.
  5. Smoke-test: each server only exposes tools from its domain.
Pros
Each turn loads only the relevant server
Easier to version independently
Cons
More processes to deploy and monitor
Initial refactor effort required
# Before — one big server apim-mcp ├── create_tenant ├── delete_tenant ├── list_apis ├── query_metrics ├── get_logs └── manage_subscription # After — split by domain apim-admin-mcp ├── create_tenant └── delete_tenant apim-analytics-mcp ├── query_metrics └── get_logs apim-dev-portal-mcp ├── list_apis └── manage_subscription
Pattern 02 · Lazy Loading

Deferred Tool Discovery

Start with zero tool manifests in context. Give the agent one tool: tool_search(query). It fetches manifests on demand, injected only for the next turn.

  1. Implement a tool_search MCP tool that accepts a natural language query.
  2. Wire it to a searchable index (see Pattern 03).
  3. Remove all other tools from initial context injection.
  4. Add a system prompt instruction to call tool_search before any other tool.
  5. On the next turn, inject only the returned manifests.
Pros
Near-zero manifest cost on turn 1
Scales to hundreds of tools
Cons
One extra round-trip before first tool use
Discovery accuracy depends on index quality
# System prompt snippet """ You have exactly ONE tool available: tool_search. Before calling any capability, call tool_search with a description of what you need. The relevant tools will be available next turn. tool_search("list active API subscribers") """ # Agent turn sequence Turn 1: agent → tool_search("list subscribers") returns: [list_subscriptions schema] Turn 2: agent → list_subscriptions( filter="active" )
Pattern 03 · Semantic Router

Vector-Backed Tool Registry

Keyword search is brittle. Embed each tool's description and run cosine similarity search at query time. Return only the top-k manifests.

  1. Write a rich natural-language description for each tool.
  2. Embed all descriptions (e.g., nomic-embed-text via Ollama on prod.aten).
  3. Store embeddings in pgvector (already in stack).
  4. Expose discover_tools(query, k=5) as the entry-point tool.
  5. Tune k — start at 5, raise if agent needs follow-up discovery calls.
Pros
Handles paraphrasing and synonyms
Single interface, easily versioned
Cons
All tools must be indexed before use
Needs vector infra (pgvector + Ollama)
def discover_tools( query: str, k: int = 5 ) -> list[ToolSchema]: # Embed the incoming query embedding = embed( query, model="nomic-embed-text" ) # Cosine similarity search in pgvector results = pgvector.search( embedding, k=k, table="tool_registry" ) # Return only the matched schemas return [ load_schema(r.tool_id) for r in results ]
Pattern 04 · Higher-Level Abstractions

Skills, CLI Bridges & Subagents

Replace clusters of fine-grained tools with a single higher-level interface. Three implementation flavors:

Skills Orchestrate 3–5 tools server-side. Expose as one MCP tool with structured I/O.
CLI Bridge Wrap toolset in a CLI. Expose a single run_cli(command) tool. Agent discovers capabilities via --help.
Subagents Heavy MCPs (Playwright, Figma) go inside a child agent. Parent delegates via a single delegate_to_X(task) tool.
Pros
Parent never sees heavy manifests
Subagents run pre-warmed contexts
Cons
CLI execution has security risk
Subagents add orchestration complexity
# Instead of 5 tools for user onboarding: # create_user, assign_role, send_email, # provision_tenant, log_audit_event # Expose one skill: onboard_user( name: str, email: str, role: str, tenant_id: str ) -> OnboardResult # Or one CLI bridge: run_cli("apim user onboard \ --name 'Kristen S' \ --role admin \ --tenant acme") # Or delegate to a subagent: delegate_to_portal_agent( task="Onboard Kristen as admin \ in the ACME tenant" )
Pattern 05 · Project Scoping

Per-Session Tool Sets

A single global mcp.json means a developer debugging an integration also has Playwright, Figma, and DB admin tools in context. Irrelevant noise — and attack surface.

  1. Create a base config with discovery-only tools.
  2. Create scoped configs per project or engagement.
  3. Pass the config at session start: --mcp-config mcp.apim.json
  4. Document which config maps to which project in the team wiki.
  5. Review and prune configs quarterly as tools change.
Pros
Zero irrelevant tools per session
Easy to audit agent access per task
Cons
Requires discipline to pick right config
Configs can drift from server state
# .claude/ directory layout .claude/ mcp.default.json ← discovery + utils mcp.apim.json ← APIM admin tools mcp.analytics.json ← metrics + logs mcp.infra.json ← Docker/SSH/systemd mcp.acme.json ← ACME engagement # Launch with scoped config claude --mcp-config .claude/mcp.apim.json # Or set per-project in project config # .claude/settings.json: { "mcpConfig": ".claude/mcp.apim.json" }
Safety & Security

These Patterns Increase Power. Guard It.

Low-context patterns often introduce execution interfaces that, if misconfigured, carry serious risk. Three things to get right before shipping.

CLI / Execute Gateway
A run_cli tool running as your user can perform destructive operations. Run CLI bridges as a dedicated low-privilege service account. Allowlist permitted commands. Log every invocation with prompt context.
Sandbox Choice
If using a code-mode executor, choose carefully. quickjs-emscripten is safer for adversarial inputs — fewer known escape paths. Avoid RestrictedPython for any untrusted content.
Prompt Injection via Results
Tool results containing attacker-controlled text can hijack routing. Strip or escape all result content before it re-enters the agent loop as context. Treat tool results as untrusted data.
Summary

Implementation Checklist

Work through these in order — each layer reinforces the next.

  • Split large MCP servers into ≤4 semantic units per component
  • Add a tool_search gateway; remove upfront manifest injection
  • Back the registry with pgvector + nomic-embed-text
  • Replace fine-grained clusters with Skills, CLI bridges, or Subagents
  • Create per-project mcp.*.json files; document which maps to what
  • Restrict CLI/execute tools to a least-privilege service account
  • Choose quickjs-emscripten over RestrictedPython for code-mode sandbox
  • Sanitize all tool results before they re-enter the agent loop
